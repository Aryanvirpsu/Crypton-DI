 1. ARCHITECTURE OVERVIEW
================================================================================

  evaluate()
    ├── Step 1: evaluate_hard_gates()  → immediate DENY if any fire
    ├── Step 2: compute_score()        → R = (D+S+N+B) * M_action * M_org + A
    ├── Step 3: degraded_signals()     → force CHALLENGE on sensitive if Redis down
    ├── Step 4: org_graph_bias()       → additive bias + threshold adjustment
    ├── Step 5: score_to_decision()    → map effective score to Decision
    ├── Step 6: escalation_memory()    → promote decision from recent history
    ├── Step 7: compute_directives()   → lock, notify, hold reason
    └── Step 8: apply_side_effects()   → lock_account, block_ip

  Constraints:
  - Zero dependencies on Crypton service code
  - All signal I/O abstracted behind traits (SignalStore, IncrementStore, OrgGraphStore)
  - Fully synchronous scoring logic — async only at signal-fetch boundaries
  - No ML, no probabilistic models. Every outcome is traceable
  - Fails open when org data is unavailable

 2. CORE FORMULA
================================================================================

  R = ((D + S + N + B) * M_action * M_org) + A

  Where:
    D  = Device trust score        (0–25)
    S  = Session anomaly score     (0–25)
    N  = Network risk score        (0–20)
    B  = Behavioral velocity score (0–20)
    Base = D + S + N + B           (0–90)

    M_action = Action sensitivity multiplier  (1.0–3.0)
    M_org    = Org risk-level multiplier      (1.0–2.0)

    Scaled = floor((Base * M_action * M_org), clamped 0–255)

    A  = Absolute adder            (0–30, applied after multiplication)

    Final Score = min(Scaled + A, 100)

  Then: org-graph bias + threshold shift → effective score (0–100)
    after_bias = min(final_score + total_org_bias, 100)
    effective_score = clamp(after_bias + threshold_shift, 0, 100)

 12. ORG-GRAPH INTELLIGENCE LAYER
================================================================================

  The org-graph detects coordinated attacks across users within a tenant.
  Graph updates happen AFTER evaluate() (fire-and-forget). Background
  recomputation rebuilds cluster membership periodically.

  ── Org Bias Computation ──

  compute_org_bias(org_risk_score) → 0–10:
    Score  0–30:  0 points
    Score 31–60:  (score - 30) / 6   (linear 0–5)
    Score 61–80:  5 + (score - 60)/7  (linear 5–8)
    Score 81–100: 8 + (score - 80)/10 (linear 8–10)

  compute_cluster_bias(membership) → 0–15:
    bias = floor(cluster_risk * confidence * severity / 10.0)

  total_bias = min(org_bias + cluster_bias, 25)  ← hard cap

  ── Threshold Shift ──

  Per-tenant adaptive threshold from TenantRiskProfile:
    Bounded [-15, +20]. Moves at most 3 points per recomputation cycle.

    attack_pressure > 0.7 → target +15 (severe)
    attack_pressure > 0.5 → target +10 (moderate)
    attack_pressure > 0.3 → target +6  (minor)
    cluster_density > 0.15 → additional +4
    stability > 0.9 && pressure < 0.1 → target -5 (proven stable)

  ── Graph Model ──

  Node types: User, Credential, IpAddress, Device
  Edge types (with base weights):
    SharedIp        (0.6) — same IP, multiple users
    SharedDevice    (0.9) — same device fingerprint
    LoginSequence   (0.5) — user A then B from same IP
    RecoveryChain   (0.8) — recovery linking between users
    ActionLink      (0.4) — both targeted same resource

  Time decay: w(t) = initial_weight * 2^(-(elapsed_hours / 6))
    Half-life: 6 hours. Pruned below 0.1.

  Memory bounds: MAX_NODES_PER_TENANT = 50,000

  ── Attack Type Classification ──

  ┌────────────────────────┬──────────┬────────────┬──────────────────────────┐
  │ Type                   │ Severity │ Base Risk  │ Classification Rule      │
  ├────────────────────────┼──────────┼────────────┼──────────────────────────┤
  │ SessionHijackCluster   │   1.5    │    80      │ ≥2 SharedDevice edges    │
  │ RecoveryAbuse          │   1.4    │    75      │ ≥2 RecoveryChain edges   │
  │ CredentialStuffing     │   1.2    │    70      │ ≥4 SharedIp AND ≥2 Login │
  │ SybilFarm              │   1.0    │    60      │ ≥3 LoginSeq AND ≥1 Actio │
  │ ProxyNetwork           │   0.7    │    45      │ ≥6 SharedIp (no other)   │
  │ UnknownAnomaly         │   0.6    │    35      │ Catch-all                │
  └────────────────────────┴──────────┴────────────┴──────────────────────────┘

  OrgRiskSnapshot staleness threshold: 5 minutes (300 seconds)

================================================================================
 13. ESCALATION MEMORY
================================================================================

  Per-user decision history stored at risk:recent:{uid} (Redis list, TTL 5min).
  Implemented via check_and_record_escalation() for TOCTOU safety.

  Rules (applied in order, 5-minute window):
    ≥ 2 Hold   decisions in 5m AND current ∈ {Challenge, Hold} → Deny
    ≥ 3 Challenge decisions in 5m AND current == Challenge      → Hold

  Production implementations MUST use Redis Lua scripts or MULTI/EXEC
  for atomicity. The default trait impl is non-atomic (get→check→record).

================================================================================
 14. CHALLENGE ACTION SELECTION LOGIC
================================================================================

  When decision == Challenge, the engine selects the appropriate friction:

    1. score >= 50            → RequiredAction::ReLogin
    2. JWT fingerprint stored
       AND stored != current  → RequiredAction::ReverifyDevice
    3. otherwise              → RequiredAction::StepUpWebAuthn

  Additional RequiredAction variants (for adapter-level use):
    4. CompleteCaptcha  — bot-suspected traffic, progressive friction
    5. VerifyEmail      — email verification needed
    6. AdminApproval    — async human review (used with Hold)

================================================================================
 15. SIDE-EFFECT DIRECTIVES
================================================================================

  lock_account:
    Set login:locked:{uid} (TTL 3600s) when:
      decision == Deny AND action ∈ {Login, OauthComplete, Register}

  block_ip:
    Set ip:blocked:{ip} (TTL 3600s) when:
      decision == Deny AND action == Register
    (Anti-sybil: prevents repeated registration from same IP)

  notify_user:
    True for any blocking decision (Challenge, Hold, Deny)

  notify_admin:
    True for Hold or Deny

  hold_reason:
    Format: "Score threshold exceeded. Top signals: {top3}. Action: {label}."
    Or:     "Escalated from repeated challenges. Top signals: {top3}. Action: {label}."

================================================================================
 22. ATTACK GRAPH MODEL
================================================================================

  Nodes:
    user:{uuid}              — authenticated user
    cred:{uuid}              — WebAuthn credential
    ip:{x.x.x.x}            — IP address
    dev:{fingerprint_hash}   — device fingerprint

  Edges (undirected, time-decaying):
    SharedIp        (0.6)  — multiple users from same IP
    SharedDevice    (0.9)  — same device fingerprint, multiple users
    LoginSequence   (0.5)  — sequential logins from same IP
    RecoveryChain   (0.8)  — recovery linking
    ActionLink      (0.4)  — both targeted same resource

  Decay: w(t) = initial_weight * 2^(-(elapsed_hours / 6))
  Half-life: 6 hours
  Pruned below: 0.1
  Reinforcement: 0.7 * base_type + 0.3 * current

  Clustering:
    Union-find with path compression + union-by-rank
    Only merges on edges with weight >= 0.25
    Clusters pruned if confidence < 0.2 or size < 2

  Confidence formula:
    confidence = w_size * size_factor + w_density * edge_density + w_weight * avg_weight
    Where weights are per-attack-type (all sum to 1.0)

  Risk score formula:
    base = attack_type.base_risk_score()
    size_factor = min(1 + ln(member_count), 4.5) / 4.5
    raw = base * confidence * size_factor
    Clamped to 0–100

================================================================================
 23. TENANT RISK PROFILES & ADAPTIVE THRESHOLDS
================================================================================

  TenantRiskProfile:
    - attack_pressure:      blocked_requests / total_requests (0.0–1.0)
    - stability_score:      1.0 - instability (0.0–1.0)
    - cluster_density:      clustered_users / total_users (0.0–1.0)
    - active_cluster_count: count of active clusters
    - threshold_shift:      [-15, +20] bounded, inertia ±3 per cycle

  Under-attack detection:
    attack_pressure > 0.6 OR active_cluster_count >= 5

  Stable tenant detection:
    stability > 0.9 AND active_clusters == 0

  OrgRiskSnapshot:
    - org_risk_score:       0–100 aggregate
    - cluster_severity_distribution: [low, medium, high, critical]
    - Stale after 5 minutes (300 seconds)
